Git Branch Strategy (2)

2019-04-08
Git

기본 브랜치

1. master (local/remote)

  • 배포(push)용 브랜치
  • 배포 준비된 코드 (stable 한 상태)
  • 주기적으로 upstream/master pull 하여 저장소의 최신 내용을 내려 받음
  • 배포(push) 준비가 완료되면 remote(origin)/master 브랜치로 push

2. develop (local)

  • 개발용 브랜치
  • 배포하기 위해 개발하는 코드는 모두 이곳에서 작업
  • 개발 완료 후 master 브랜치로 병합(merge)
  • 해당 브랜치는 로컬 저장소에 상주하며, master 브랜치의 내용을 병합(merge)하여 최신본을 유지

보조 브랜치 (optional)

1. feature/~ (local/remote)

  • 브랜치 운용 정책에 따라 프로젝트 담당 개발자는 협업용 브랜치를 생성한다.
  • 마크업이 완료되어 퍼블리셔가 push한 내용이 담기면 develop 브랜치로 merge
  • 프로젝트가 완전히 종료되면 해당 브랜치는 삭제한다.

2. hotfixes/~ (local)

  • 계획되지 않은 긴급 배포를 위한 브랜치
  • QA 중 수정사항이 나온 경우에도 해당 브랜치를 사용하여 작업하면 된다.
  • 가장 최신 소스인 master 브랜치로부터 분기
  • master, develop에 병합 후 삭제한다.

작업본 운용 방침 (personal)

  1. 히스토리 라인은 최대한 간결하게 가져간다.
  2. pull을 자주하여 저장소와 비슷한 이력을 유지한다. (최신 코드 유지)
  3. commit은 의미있는 단위로 하되 모든 동작이 기능하는 상태여야 한다. (stable)
